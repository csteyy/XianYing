## 二、推荐的几种方式（从最推荐到可选）

### 方案 1：**在每个文件夹放一个 `_INDEX.md`**（最推荐）

在关键文件夹里放一个 `_INDEX.md`（下划线开头确保它排在文件列表最前面），用简洁的表格说明该目录下每个文件的职责。例如在 `src/components/` 下：

```markdown
# components/ 目录索引

本目录包含显·影 App 的所有页面组件和功能组件。

## 页面组件（对应 App 中的一个完整页面）
| 文件 | 对应页面 | 简述 |
|---|---|---|
| HomePage.tsx | 首页 | 主入口，粒子背景 + 功能卡片 |
| RecordingPage.tsx | 录制页 | 实时录音 + 转录 + 波形 |
| DataAnalysisPage.tsx | 数据分析 | 互动统计 + 气氛转折 |
| ... | ... | ... |

## 功能组件（被页面复用的小模块）
| 文件 | 简述 |
|---|---|
| BottomNav.tsx | 底部 Tab 导航栏 |
| FloatingActionButton.tsx | 新建记录的圆形按钮 |
| ParticleBackground.tsx | 首页装饰性粒子动画 |
```

**优点**：不污染代码文件，打开文件夹就能看到目录说明，维护成本低。

### 方案 2：**用 Cursor Rule 让 AI 自动在每次改动后更新说明**

你可以创建一个 `.cursor/rules/` 规则文件，告诉 AI 每次新增或修改组件时，自动在文件头部加一段标准化注释。例如：

```typescript
/**
 * @page 录制页
 * @role 核心功能页面
 * @description 实时录音、语音转写字幕、音频波形可视化
 * @flow 场景设置 → [本页] → 录后编辑
 * @depends SoundBars, speakerColors
 */
```

这样每个文件打开第一眼就能看到它在产品流程中的位置。

### 方案 3：**建一个全局的架构地图文档**

在项目根目录维护一个 `ARCHITECTURE.md`，用流程图 + 表格描述整体架构。内容类似我上面给你列的表格，但加上页面间的跳转关系：

```
首页 (HomePage)
  ├─→ 场景设置 (RecordingModeSelectionPage)
  │     └─→ 录制 (RecordingPage)
  │           └─→ 录后编辑 (PostRecordingEditPage)
  │                 └─→ 数据分析 (DataAnalysisPage)
  │                       └─→ 可视化 (Visualization3DPage) → 回到首页
  ├─→ 历史记录 (HistoryPage)
  ├─→ 作品库 (GalleryPage)
  └─→ 设置 (SettingsPage)
```

### 方案 4：**文件命名加前缀标记**（你提到的方式）

比如把 `RecordingPage.tsx` 改名为 `[页面]RecordingPage.tsx`，`BottomNav.tsx` 改为 `[导航]BottomNav.tsx`。

**我不太推荐这种方式**，原因是：
- 会破坏代码中所有 import 路径，需要大量联动修改
- 中文文件名在某些构建工具中可能出问题
- 和社区惯例差距大，未来协作会增加认知负担
